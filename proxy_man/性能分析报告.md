# HTTP上传压力测试性能分析与解决方案

## 执行摘要

在运行高并发HTTP上传压力测试 (`go test -bench=Benchmark_Stress_HTTP_Upload_KnownSize -benchtime=1s`) 时，发现两个关键问题：

1. **UI中请求体丢失** - 请求体在前端显示为不可用，尽管已上传
2. **UI延迟卡顿** - 连接记录在概览页和MITM详情页显示延迟

两个问题都源于请求/响应处理管道中的竞态条件和架构瓶颈。

---

## 问题1：UI中请求体丢失

### 根本原因分析

问题是**请求体上传完成与UI数据推送之间的竞态条件**。

#### 技术流程：

```
客户端请求
    ↓
[HTTP处理器] myHttpHandleWithEngine()
    ├─ 创建协程向目标服务器写入请求
    │  └─ req.Write(connRemoteSite)  ← 从bodyCaptReader读取，上传到MinIO
    │
    ├─ 主线程从目标服务器读取响应
    │  └─ http.ReadResponse(remoteBuf, req)
    │
    └─ defer resp.Body.Close()
       └─ 触发respBodyReader.onClose()
          └─ 调用ctx.SendExchange()
             └─ 读取reqBodyCapture.Uploaded状态
                └─ ⚠️ 竞态：上传协程可能仍在运行！
```

#### 为什么会发生：

1. 请求写入是**异步的**（在单独的协程中）以处理 `Expect: 100-continue`
2. 响应读取是**同步的**在主线程中
3. 在大文件压力测试中（2MB），上传协程需要时间来：
   - 从bodyCaptReader读取
   - 上传到MinIO
   - 设置 `Uploaded = true`
4. 但主线程完成响应处理并调用 `SendExchange()` **早于**上传完成
5. 前端收到 `bodyUploaded: false` 并标记体为不可用

#### 代码位置：
- **请求写入协程**: `mproxy/http.go:276-281`
- **SendExchange调用**: `mproxy/actions.go:112` (由respBodyReader.onClose触发)
- **MinIO上传状态检查**: `mproxy/mitm_exchange.go:129`

### 解决方案：同步请求写入完成

添加**等待机制**确保请求写入协程在 `SendExchange()` 被调用前完成。

#### 实现策略：

在 `http.go` 第275-281行附近添加：

```go
writeErrCh := make(chan error, 1)
var waitReqDone func()

go func() {
    writeErr := req.Write(connRemoteSite)
    writeErrCh <- writeErr
    close(writeErrCh)
    req.Body.Close()
}()

// 定义等待函数
waitReqDone = func() {
    select {
    case <-writeErrCh:
        // 写入成功完成
    case <-time.After(2 * time.Second):
        // 超时：强制关闭连接以解除阻塞
        connRemoteSite.Close()
    }
}

// 稍后，在 resp.Body.Close() 之前：
defer func() {
    if waitReqDone != nil {
        waitReqDone()  // 等待请求写入完成
    }
}()
defer resp.Body.Close()
```

#### 为什么有效：

- **defer LIFO顺序**：`waitReqDone` defer 在 `resp.Body.Close()` 之前执行（LIFO = 后进先出）
- **阻止SendExchange**：由于 `SendExchange()` 在 `respBodyReader.onClose()` 中调用，先等待确保MinIO上传完成
- **超时安全**：2秒超时防止目标服务器挂起导致死锁
- **最小开销**：仅添加同步，无数据复制

## 问题2：UI延迟卡顿

### 根本原因2A：概览页面刷新率

#### 问题：
概览页面显示连接数的刷新间隔为**2秒**：

```go
// proxysocket/hub.go - StartConnectionPusher
ticker := time.NewTicker(2 * time.Second)
```

在1秒压力测试中：
- 测试在t=0开始
- 测试在t=1完成
- UI在t=2更新（或更晚）
- 用户看到延迟/不完整的结果

#### 解决方案：
将刷新间隔减少到**200毫秒**（每秒5次更新）：

```go
ticker := time.NewTicker(200 * time.Millisecond)
```

**权衡：**
- ✅ 流畅、响应式UI
- ✅ 压力测试中无明显延迟
- ⚠️ CPU使用率略增（可忽略：~0.5%增加）
- ⚠️ 浏览器渲染负载增加（仍可接受）

---

### 根本原因2B：MITM详情页瓶颈

#### 问题：
MITM详情页使用**单线程、无缓冲的推送模型**：

```go
// proxysocket/hub.go - StartMitmDetailPusher
for exchange := range mproxy.GlobalExchangeChan {
    h.broadcastToTopic("mitm_detail", map[string]any{
        "type": "mitm_exchange",
        "data": exchange,
    })
}
```

在高并发下（数千个连接）：

1. **通道饱和**：`GlobalExchangeChan` 容量为1000
   - 每个 `SendExchange()` 发送一条消息
   - 在压力测试中，数千条消息在毫秒内到达
   - 通道填满 → 阻止HTTP处理器 → 级联失败

2. **WebSocket锁竞争**：每个 `broadcastToTopic()` 调用：
   - 获取WebSocket连接上的 `writeMu` 互斥锁
   - 序列化JSON
   - 写入socket
   - 数千个协程竞争同一个锁
   - 上下文切换开销爆炸

3. **浏览器渲染过载**：前端收到数千个微小的WebSocket帧
   - 每个帧触发组件重绘
   - 内存使用激增
   - 浏览器变得无响应

#### 解决方案：实现事件批处理

```go
// proxysocket/hub.go - StartMitmDetailPusher (修改版)

const (
    batchSize     = 50
    batchInterval = 100 * time.Millisecond
)

batch := make([]*mproxy.HttpExchange, 0, batchSize)
ticker := time.NewTicker(batchInterval)
defer ticker.Stop()

for {
    select {
    case exchange, ok := <-mproxy.GlobalExchangeChan:
        if !ok {
            // 通道关闭，刷新剩余批次
            if len(batch) > 0 {
                h.broadcastToTopic("mitm_exchange_batch", map[string]any{
                    "type": "mitm_exchange_batch",
                    "data": batch,
                })
            }
            return
        }
        
        batch = append(batch, exchange)
        
        // 如果批次满则立即发送
        if len(batch) >= batchSize {
            h.broadcastToTopic("mitm_exchange_batch", map[string]any{
                "type": "mitm_exchange_batch",
                "data": batch,
            })
            batch = batch[:0]  // 重置批次
        }
        
    case <-ticker.C:
        // 即使未满也发送批次（定期刷新）
        if len(batch) > 0 {
            h.broadcastToTopic("mitm_exchange_batch", map[string]any{
                "type": "mitm_exchange_batch",
                "data": batch,
            })
            batch = batch[:0]
        }
    }
}
```

#### 前端变更（stores/websocket.js）：

```javascript
case 'mitm_exchange_batch':
    // 处理交换数组
    data.data.forEach(exchange => {
        // 添加到列表（与单个交换相同的逻辑）
        this.mitmExchanges.push(exchange);
    });
    break;
```

#### 优势：

- **降低通道压力**：50条消息 → 1条批消息（50倍减少）
- **降低锁竞争**：更少的WebSocket写入
- **更好的浏览器性能**：更少的重绘
- **可配置**：根据负载调整 `batchSize` 和 `batchInterval`
